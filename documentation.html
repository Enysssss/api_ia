NHANES — une source de données clinique réelle

NHANES (National Health and Nutrition Examination Survey) est une enquête nationale conduite par le CDC américain depuis 1960. Des milliers de volontaires se rendent dans des centres d'examen mobiles et passent des batteries de tests médicaux standardisés.

Pourquoi NHANES plutôt qu'un dataset synthétique

❌ Données synthétiques (ex: generate_dataset.py)

Les features sont tirées de distributions gaussiennes indépendantes. Aucune corrélation naturelle. Un IMC de 30 n'implique rien sur le % de graisse. Les patterns appris n'existent pas dans le monde réel. Le modèle ne généralisera jamais sur de vraies personnes.

✓ NHANES (mesures cliniques)

Vraies corrélations : l'IMC et le % de graisse sont naturellement liés, l'âge influe sur la composition corporelle, la perception du poids dépend de l'état réel. Ces relations complexes sont ce que le modèle apprend vraiment.

Les 4 fichiers XPT — format SAS

NHANES distribue ses données en fichiers XPT (format SAS XPORT). pandas les lit nativement avec pd.read_sas(). Chaque fichier couvre un type de mesure différent mais contient le même identifiant de participant : SEQN.

FichierContenuParticipantsColonne utilisée
DEMO_J.xptDonnées démographiques9 254RIDAGEYR — âge en années
BMX_J.xptMesures corporelles (anthropométrie)8 704BMXHT (taille cm) + BMXWT (poids kg)
DXX_J.xptAbsorptiométrie bi-photonique (DXA) corps entier5 114DXDTOPF — Total Body Fat %
WHQ_J.xptQuestionnaire historique du poids6 161WHQ030 — perception du poids
Qu'est-ce que la DXA ?

La Dual-Energy X-ray Absorptiometry est un scan médical qui mesure la composition corporelle avec une précision de 1-2%. Elle différencie la masse grasse, la masse maigre et la densité osseuse. C'est la méthode de référence en médecine du sport — bien plus précise que les balances à impédance. DXDTOPF donne le pourcentage de graisse sur l'ensemble du corps.

Fusion des 4 fichiers — merge sur SEQN

Chaque participant NHANES reçoit un identifiant unique SEQN présent dans tous les fichiers. C'est la clé qui permet d'assembler les informations dispersées en une seule ligne.

# Chargement des 4 sources indépendantes
demo = pd.read_sas("DEMO_J.xpt")[["SEQN", "RIDAGEYR"]]
bmx  = pd.read_sas("BMX_J.xpt") [["SEQN", "BMXHT", "BMXWT"]]
dxx  = pd.read_sas("DXX_J.xpt") [["SEQN", "DXDTOPF"]]
whq  = pd.read_sas("WHQ_J.xpt") [["SEQN", "WHQ030"]]

# Jointure en cascade — inner join implicite
# Une ligne disparaît dès qu'un fichier n'a pas ce SEQN
merged = demo.merge(bmx, on="SEQN") \
             .merge(dxx, on="SEQN") \   # ← perd ~4000 lignes (DXA partiel)
             .merge(whq, on="SEQN")

Pourquoi on perd autant de lignes

ÉtapeLignes restantesRaison

+ BMX~8 600Quelques personnes n'ont pas pu être pesées/mesurées

Le merge est une inner join par défaut

pandas .merge() sans paramètre how= fait une inner join : on ne garde que les SEQN présents dans les deux tables. Un SEQN absent de DXX_J disparaît du résultat, même s'il est dans tous les autres fichiers. C'est le comportement voulu — on veut des lignes complètes.

Conception du label

La variable WHQ030 est la réponse à une question posée lors d'un entretien médical conduit par un enquêteur CDC formé. C'est une variable psycho-sociale indépendante des mesures biométriques.

"How do you consider your current weight ?"

Pourquoi WHQ030 est un bon label ML

  • Source indépendante : cette réponse vient de la perception du participant, influencée par ses antécédents médicaux, les conseils de son médecin, sa culture, son âge — pas seulement son IMC instantané.
  • Non-déductible mécaniquement : un individu avec IMC=26 peut répondre "overweight" ou "about right" selon son contexte. Le modèle doit apprendre cette nuance, il ne peut pas la calculer.
  • Réponse cliniquement pertinente : c'est exactement ce qu'un système de recommandation fitness doit prédire — non pas un seuil arbitraire, mais l'intervention que la personne considère elle-même nécessaire.

Data leakage — le piège évité

Le data leakage est l'erreur la plus insidieuse en ML. Le modèle obtient une accuracy artificiellement parfaite mais n'a rien appris de généralisable.

Définition

Le data leakage survient quand les features contiennent directement ou indirectement l'information du label. Le modèle apprend à "tricher" en lisant la réponse dans les entrées plutôt qu'en apprenant un vrai pattern.

Exemple concret de leakage (ce qui était faux avant)

# ❌ LEAKAGE : label calculé depuis les features elles-mêmes
if imc > 27 or fat > 33:    label = "perte_poids"
if imc < 18.5 or fat < 10: label = "prise_masse"
else:                        label = "maintien"

# Puis on entraîne avec les mêmes variables :
X = [[imc, fat, age]]   # ← imc et fat sont dans les règles du label !

# Résultat : accuracy 99.96% → le modèle a mémorisé 3 lignes de if/else

GroupeVariablesSource dans NHANESContribue au label ?
Featuresimc, taux_masse_grasse, ageBMX_J (taille/poids), DXX_J (DXA), DEMO_JNon
Labelperte_poids / maintien / prise_masseWHQ_J (perception déclarée)—
Séparation totale des sources

Les features viennent de mesures physiques instrumentales (scan DXA, balance, toise). Le label vient d'un entretien verbal. Ces deux types de données ont été collectés indépendamment, dans des conditions différentes, par des opérateurs différents. Aucune règle ne relie mécaniquement les features au label.

Comment détecter un leakage

  • Accuracy > 95% sur un problème biologique/humain → suspect
  • Les importances de features correspondent exactement aux variables utilisées pour générer les labels
  • Le modèle performe parfaitement sur des exemples qu'un humain ne pourrait pas prédire
  • En production : les prédictions sont toujours correctes même sur des profils inhabituels
Machine Learning

Encodage des labels — LabelEncoder

Un algorithme ML ne manipule que des nombres flottants. Les labels texte doivent être convertis en entiers avant tout calcul.

Ce que fait LabelEncoder concrètement

Il scanne tous les labels uniques du dataset, les trie alphabétiquement, et leur assigne un entier croissant. L'ordre est déterministe et reproductible :

"maintien"alphabétique : m
→
0
"perte_poids"alphabétique : p-e
→
1
"prise_masse"alphabétique : p-r
→
2
from sklearn.preprocessing import LabelEncoder

encoder = LabelEncoder()
y_encoded = encoder.fit_transform(y)
# ["maintien","perte_poids","maintien","prise_masse"] → [0, 1, 0, 2]

# Opération inverse — utilisée à la prédiction
encoder.inverse_transform([1])   # → ["perte_poids"]

# Voir la correspondance complète
list(zip(encoder.classes_, range(len(encoder.classes_))))
# → [("maintien", 0), ("perte_poids", 1), ("prise_masse", 2)]

Pourquoi pas OneHotEncoder ?

OneHotEncoder crée une colonne binaire par classe (3 colonnes ici). C'est nécessaire pour les modèles linéaires qui interpréteraient 0 < 1 < 2 comme une relation d'ordre. Le RandomForest n'a aucune hypothèse d'ordre — il partitionne par seuils. LabelEncoder est donc suffisant et plus simple.

Pourquoi sauvegarder encoder.pkl

Le piège classique :

Recréer un LabelEncoder à la prédiction avec fit_transform() sur les labels disponibles à ce moment-là peut produire une correspondance différente si une classe est absente du batch. Il faut toujours recharger l'objet exact de l'entraînement. encoder.pkl est la mémoire permanente de cette correspondance.

Machine Learning

Split train/test — évaluation honnête

Pourquoi diviser le dataset

Si on entraîne et évalue sur les mêmes données, le modèle peut mémoriser les exemples sans rien généraliser. Les métriques seraient fausses. On réserve 20% des données — que le modèle ne verra jamais pendant l'entraînement — pour mesurer la vraie performance.

X_train, X_test, y_train, y_test = train_test_split(
    X, y_encoded,
    test_size=0.2,       # 80% entraînement, 20% test
    random_state=42,     # graine aléatoire → résultats reproductibles
    stratify=y_encoded   # ← crucial pour les classes déséquilibrées
)
# Train : 1 947 lignes  |  Test : 487 lignes

Pourquoi stratify est indispensable ici

prise_masse représente seulement 5.8% du dataset (140 lignes). Sans stratify, un split purement aléatoire pourrait mettre 130 de ces 140 lignes dans le train set et seulement 10 dans le test.

LabelTotalTrain (80%)Test (20%)Sans stratify (risque)
perte_poids1 229983246~246 ± quelques unités
maintien1 065852213~213 ± quelques unités
prise_masse14011228⚠️ 5 à 40 selon la chance
Avec stratify, chaque classe garde ses proportions dans train ET test.

Le recall, la precision et le F1 de prise_masse sont mesurés sur 28 exemples représentatifs, pas sur un échantillon biaisé par hasard.

Machine Learning

RandomForestClassifier — le modèle choisi

Comment fonctionne un arbre de décision

Un arbre de décision partitionne l'espace des features en posant des questions binaires successives. À chaque nœud, il choisit la feature et le seuil qui réduisent le plus l'impureté (mélange de classes) dans les deux sous-groupes résultants.

# Exemple d'arbre simplifié appris sur nos données :
if imc > 29.5:
    if taux_masse_grasse > 38.0:
        → perte_poids  (95% des cas à cette feuille)
    else:
        → perte_poids  (78% des cas)
else:
    if taux_masse_grasse < 18.5:
        if age < 35:
            → prise_masse (62% des cas)
        else:
            → maintien    (71% des cas)
    else:
        → maintien        (69% des cas)

De l'arbre à la forêt — le principe du bagging

Un seul arbre overfitte facilement (il mémorise les exemples d'entraînement). La solution : entraîner 100 arbres sur des sous-échantillons légèrement différents du dataset (bootstrap), puis faire voter tous les arbres.

Ce que les importances nous apprennent

IMC (50%) + % graisse (33%) = 83% du pouvoir prédictif. Cohérent avec la médecine : la composition corporelle prédit la perception du poids. L'âge contribue 17% — bien plus que sur un dataset synthétique. Pourquoi 76.6% et pas 99% ?

76.6% est une accuracy honnête et défendable.

La perception qu'une personne a de son propre poids (WHQ030) ne se déduit pas mécaniquement de sa biométrie. Deux personnes avec les mêmes IMC et % de graisse peuvent répondre différemment selon leur médecin, leur culture, leur historique. Cette ambiguïté réelle produit une accuracy réaliste. Un modèle à 99% ici serait suspect — il indiquerait presque certainement un leakage.

Machine Learning

Sérialisation — model.pkl & encoder.pkl

Qu'est-ce que la sérialisation

Un objet Python (RandomForestClassifier entraîné, LabelEncoder) existe en mémoire RAM pendant l'exécution du script. Quand le script se termine, il disparaît. La sérialisation consiste à convertir cet objet en une suite d'octets sauvegardée sur disque — le fichier .pkl. À la prédiction, on désérialise (recharge) l'objet dans l'état exact où il était à la fin de l'entraînement.

import joblib

# À l'entraînement — sauvegarde
joblib.dump(model,   "models/model.pkl")
joblib.dump(encoder, "models/encoder.pkl")

# Au démarrage de l'API — rechargement
model   = joblib.load("models/model.pkl")
encoder = joblib.load("models/encoder.pkl")

joblib vs pickle — pourquoi joblib

joblib est spécialisé pour les objets numpy (arrays multidimensionnels). Un RandomForest entraîné contient des centaines de tableaux numpy (les seuils de chaque nœud de chaque arbre). joblib les compresse et les sérialise bien plus efficacement que le module pickle standard.

Pourquoi deux fichiers séparés

FichierContenuUtilisé quand
model.pklLes 100 arbres avec tous leurs nœuds et seuilsÀ chaque prédiction — model.predict()
encoder.pklLa correspondance entier ↔ label texteAprès prédiction — encoder.inverse_transform()

Les séparer respecte le principe de responsabilité unique. Si on change le mapping des labels sans réentraîner, on peut remplacer encoder.pkl seul. Si on réentraîne le modèle, le mapping reste stable.

Pydantic valide chaque champ entrant avant d'appeler le service. Un seul champ invalide retourne automatiquement une erreur HTTP 422 avec le détail précis du problème.

API FastAPI

Singleton Service — chargement unique

Le problème sans singleton

Charger un fichier .pkl depuis le disque implique : accès disque (I/O), désérialisation des octets, reconstruction des objets numpy. C'est une opération qui prend ~200ms à ~2s selon la taille du modèle. Si on recharge à chaque requête HTTP, l'API est 100x plus lente.

Le pattern singleton

class NutritionService:
    _instance = None        # attribut de CLASSE — partagé par toutes les instances

    def __new__(cls):
        if cls._instance is None:   # premier appel seulement
            cls._instance = super().__new__(cls)
            cls._instance._model   = joblib.load("models/model.pkl")
            cls._instance._encoder = joblib.load("models/encoder.pkl")
        return cls._instance          # tous les appels suivants → même objet

# Dans main.py — exécuté une fois au démarrage du serveur
service = NutritionService()

# À chaque requête — instantané, le modèle est déjà en RAM
NutritionService()  # retourne le même objet, pas de rechargement
Résultat :

Le chargement des .pkl se produit une seule fois au démarrage de uvicorn. Toutes les requêtes suivantes réutilisent les objets déjà en mémoire. Latence de prédiction : quelques millisecondes au lieu de plusieurs secondes.

API FastAPI

Prédiction — pas à pas

def predict(self, data: NutritionInput) -> NutritionOutput:

    # ① Calcul IMC depuis les données brutes de l'utilisateur
    imc = data.poids_kg / (data.taille_cm / 100) ** 2
    #    ex: 90 / (1.75)² = 29.39 kg/m²

    # ② Construction du vecteur features
    #    ORDRE IDENTIQUE à FEATURES dans train_models.py
    #    ["imc", "taux_masse_grasse", "age"]
    X = [[imc, data.taux_masse_grasse, data.age]]

    # ③ Prédiction — les 100 arbres votent
    encoded_pred = self._model.predict(X)[0]   # → entier ex: 1

    # ④ Décodage — retrouve le label texte
    label = self._encoder.inverse_transform([encoded_pred])[0]  # → "perte_poids"

    # ⑤ Probabilités — ratio de votes des 100 arbres
    probas     = self._model.predict_proba(X)[0]
    #    ex: [0.25, 0.70, 0.05]  →  maintien=25% / perte=70% / prise=5%
    confidence = float(np.max(probas))          # → 0.70

    # ⑥ Assemblage de la réponse
    return NutritionOutput(
        imc          = round(imc, 2),
        imc_categorie = _imc_categorie(imc),   # seuils OMS
        label        = label,
        confidence   = round(confidence, 4),
        message      = _MESSAGES[label],       # conseil statique
    )
⚠️ L'ordre des features est critique et silencieux

Écrire [[age, imc, fat]] au lieu de [[imc, fat, age]] produit des prédictions fausses sans aucune erreur Python, aucune exception, aucun warning. Le modèle calcule sur les mauvaises colonnes. L'ordre doit être identique à la liste FEATURES de train_models.py.

Comment predict_proba() fonctionne en détail

Pour chaque arbre parmi les 100 :

  1. Descente dans l'arbre

    L'observation descend de nœud en nœud en suivant les conditions (ex: IMC > 29.5 → branche droite).

  2. Arrivée à une feuille

    La feuille contient la distribution des classes observées à l'entraînement pour ce sous-groupe (ex: 85% perte, 12% maintien, 3% prise).

  3. Moyenne sur 100 arbres

    Les distributions de toutes les feuilles atteintes sont moyennées. Résultat : [0.25, 0.70, 0.05].

  4. Classe prédite

    Celle avec la probabilité max → index 1 → "perte_poids". Confidence = 0.70.

Questions jury — réponses préparées

Pourquoi 76.6% d'accuracy ? C'est insuffisant.
76.6% est une accuracy honnête sur un problème psycho-social réel. Le label est la perception qu'une personne a de son propre poids — influencée par son médecin, sa culture, son âge. Deux individus avec des biométries identiques peuvent se percevoir différemment. Un modèle à 99% sur ce problème indiquerait presque certainement un data leakage, pas une vraie performance.
Expliquez ce qu'est le data leakage et comment vous l'avez évité.
Le data leakage survient quand les features contiennent directement l'information du label — le modèle mémorise des règles plutôt qu'apprendre. Ici, les features viennent de mesures instrumentales (DXA, balance, toise). Le label vient d'un entretien verbal (WHQ030). Ces deux types de données ont été collectés indépendamment par le CDC. Aucune formule ne calcule WHQ030 depuis l'IMC ou le % de graisse.
Pourquoi RandomForest et pas un réseau de neurones ?
Un réseau de neurones est surdimensionné pour 2 434 lignes et 3 features. Il nécessiterait un tuning extensif (nombre de couches, learning rate, dropout) pour éviter l'overfitting, sans gain garanti. RandomForest est adapté à ce volume, invariant au scaling, interprétable via les importances, et donne une confidence native via predict_proba().
Pourquoi NHANES ? Pourquoi pas générer vos propres données ?
Des données générées aléatoirement ont des distributions indépendantes sans corrélations naturelles. Un IMC de 30 synthétique n'implique rien sur le % de graisse. NHANES contient de vraies personnes mesurées cliniquement — les corrélations réelles entre âge, IMC, % graisse et perception du poids sont présentes dans les données. Le modèle apprend des patterns qui existent vraiment.
À quoi sert le LabelEncoder et pourquoi le sauvegarder ?
Un modèle ML ne traite que des nombres. LabelEncoder convertit les labels texte en entiers de façon déterministe et alphabétique. On le sauvegarde parce que la correspondance (ex: 1 = "perte_poids") doit être exactement la même à la prédiction qu'à l'entraînement. Si on recréait un encoder à la prédiction, l'ordre pourrait changer si une classe est absente du batch.
Qu'est-ce que class_weight="balanced" et pourquoi l'utiliser ?
prise_masse représente 5.8% du dataset. Sans correction, le modèle maximise l'accuracy globale en ignorant cette classe — il ne se tromperait que 5.8% du temps en prédisant toujours "pas prise_masse". class_weight="balanced" augmente automatiquement le poids des exemples des classes minoritaires dans la fonction de coût, forçant le modèle à les apprendre correctement.
Pourquoi utilisez-vous un singleton pour le service ?
Charger un fichier .pkl depuis le disque prend plusieurs centaines de millisecondes. Si on recharge le modèle à chaque requête HTTP, l'API devient 100 fois plus lente. Le singleton garantit que le chargement se produit une seule fois au démarrage du serveur. Toutes les requêtes suivantes réutilisent les objets déjà en mémoire RAM.
